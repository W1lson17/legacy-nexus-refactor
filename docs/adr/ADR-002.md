# ADR-002: Mecanismo de Autenticación mediante JWT y Cookies HttpOnly

## Context

El sistema legacy almacena contraseñas en texto plano y usa sesiones de servidor con Flask. Esto presenta múltiples problemas de seguridad: las contraseñas son vulnerables ante breaches, las sesiones no escalan horizontalmente, y no hay protección contra XSS para tokens de autenticación. Necesitamos un mecanismo de autenticación seguro, stateless y compatible con la arquitectura hexagonal.

## Options Considered

1. **JWT + HttpOnly Cookies + bcrypt**
   - Pros: Stateless — el servidor no almacena sesión. Las cookies HttpOnly + Secure son inmunes a XSS (JavaScript no puede leerlas). bcrypt tiene salt automático y es resistente a ataques de fuerza bruta. Compatible con APIs RESTful. Se integra limpiamente en hexagonal: el puerto de auth vive en `infrastructure/auth/`, los casos de uso solo ven la interfaz del puerto.
   - Cons: JWT no se puede revocar sin infraestructura extra (blacklist). La cookie HttpOnly requiere configuración CORS específica para credenciales. El acceso al JWT desde el cliente es imposible (by design) — necesitas endpoints dedicados para refresh.

2. **Sesiones de servidor (express-session + Redis)**
   - Pros: Revocación inmediata — eliminar la sesión de Redis y listo.熟知 por desarrolladores. Sin problemas de expiración de token.
   - Cons: Stateful — requiere Redis o almacenamiento compartido, lo que complica el despliegue. No escala bien para arquitecturas serverless. Acoplado a la infraestructura de sesión.

3. **OAuth2 / Proveedor de identidad externo (Auth0, Clerk, Supabase Auth)**
   - Pros: Zero código de autenticación propio. Gestión de usuarios, MFA, social login listos. Escala sin esfuerzo.
   - Cons: Dependencia de un servicio externo — si Auth0 cae, la app cae. Costo por usuario activo. El dominio de autenticación queda fuera de nuestro control. Overkill para un ERP interno con usuarios administrativos conocidos.

4. **JWT en localStorage**
   - Pros: Simple de implementar. Acceso directo desde JavaScript.
   - Cons: Vulnerable a XSS — cualquier script inyectado puede leer el token. Violación directa de OWASP. No es aceptable para producción.

## Decision

Elegimos **JWT almacenado en cookies HttpOnly + Secure + SameSite=Strict** con **bcrypt** para el hash de contraseñas.

- Los JWT se generan en `infrastructure/auth/` y se envían como cookies HttpOnly que el navegador envía automáticamente.
- El middleware de autenticación verifica el JWT en cada request protegida.
- Los passwords se hashean con bcrypt (cost factor 12) antes de almacenarse.
- Los tokens de refresh se manejan mediante un endpoint dedicado que rota el JWT.
- El puerto de autenticación (`domain/ports/auth.port.ts`) define la interfaz que los casos de uso consumen — nunca tocan JWT directamente.

## Consequences

- **Easier**: Autenticación stateless que escala horizontalmente sin Redis. Protección contra XSS (HttpOnly) y CSRF (SameSite=Strict). bcrypt es el estándar de facto para hashing de passwords.
- **Harder**: Revocación de JWT requiere una blacklist o estrategia de refresh-token cortos. CORS debe configurarse con `credentials: 'include'`. No se puede leer el token desde JavaScript para lógica de frontend — toda la info del usuario se obtiene via endpoints `/me`.
- **New constraints**: Todas las respuestas de auth deben usar cookies, nunca enviar el JWT en el body. Las cookies deben tener flags: HttpOnly, Secure (en producción), SameSite=Strict. El cost factor de bcrypt no debe bajar de 10.
- **Migration path**: Sprint 2 implementa el puerto de auth y el adaptador JWT. Sprint 3 agrega el middleware Express y los endpoints de login/logout/refresh.
