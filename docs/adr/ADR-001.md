# ADR-001: Implementación de Arquitectura Hexagonal en el Backend

## Context

El sistema legacy es un monolito Python/Flask donde las rutas HTTP mezclan lógica de negocio, acceso a base de datos y presentación. No existe separación de responsabilidades: una misma función maneja validación, persistencia y respuesta HTTP. Esto hace difícil testear, mantener y evolucionar el sistema. Necesitamos una arquitectura que proteja el dominio de negocio de las dependencias de infraestructura.

## Options Considered

1. **Arquitectura Hexagonal (Ports & Adapters)**
   - Pros: Aislamiento total del dominio — cero imports de frameworks. Los casos de uso dependen solo de interfaces (ports), nunca de implementaciones concretas. Los adaptadores (Express, Prisma) se intercambian sin tocar la capa de negocio. El ESLint `no-restricted-imports` puede enforcear las fronteras automáticamente.
   - Cons: Más archivos (barrel exports, interfaces, adaptadores). Curva de aprendizaje para desarrolladores no familiarizados con el patrón.

2. **Arquitectura en Capas (Layered)**
   - Pros: Patrón más conocido, curva de aprendizaje baja. Estructura lineal: presentación → negocio → datos.
   - Cons: Las capas tienden a filtrar dependencias hacia abajo. Sin enforcement estricto, la capa de negocio termina importando el ORM o Express. No hay mecanismo de inversión de dependencias — difícil de testear en aislamiento.

3. **Clean Architecture**
   - Pros: Similar a hexagonal con círculos concéntricos explícitos. Énfasis en la regla de dependencia.
   - Cons: Más teórica y abstracta. Sin structural enforcement en TypeScript — la regla de dependencia es solo convención. Overkill para un proyecto con 3 capas claras.

4. **Feature-Sliced Design**
   - Pros: Organización por feature en lugar de por capa. Escala bien en frontend.
   - Cons: Pensada para UI, no para backend. Pierdes el enforcement de "dominio sin frameworks". No resuelve el problema original de separar negocio de infraestructura.

## Decision

Elegimos **Arquitectura Hexagonal** con tres capas: `domain/`, `application/`, `infrastructure/`. El dominio (`domain/`) tiene cero imports de Express, Prisma o cualquier framework. Los casos de uso (`application/`) dependen solo de `domain/` y `@shared/*`. La infraestructura (`infrastructure/`) implementa los ports del dominio y orquesta los frameworks.

El enforcement se hace con ESLint `no-restricted-imports`:
- `domain/` no puede importar `express`, `@prisma/client`, ni `zod`
- `application/` no puede importar `../infrastructure/*`
- `infrastructure/` puede importar cualquier cosa

Las barrel exports (`index.ts`) en cada capa centralizan la API pública: `import { Product } from '@domain'` en lugar de rutas internas.

## Consequences

- **Easier**: Testear casos de uso con mocks de repositorios sin tocar la base de datos. Intercambiar Express por Fastify o Hono sin cambiar una línea de dominio. Agregar nuevos adaptadores sin modificar el core.
- **Harder**: Más archivos boilerplate (interfaces de ports, barrel exports). Los desarrolladores que no conocen hexagonal necesitan orientación inicial.
- **New constraints**: Toda nueva dependencia de infraestructura en `domain/` es un error de arquitectura — el linter lo detecta. Los PRs que importen frameworks en dominio deben ser rechazados.
- **Migration path**: Sprint 1 establece la estructura. Sprint 2 agrega las entidades del dominio y los casos de uso dentro de los ports ya definidos.
